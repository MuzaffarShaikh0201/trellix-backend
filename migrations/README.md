# Trellix Backend Database Migrations

This directory contains [Alembic](https://alembic.sqlalchemy.org/) database migrations for Trellix Backend. Migrations manage schema changes to the PostgreSQL database.

## Prerequisites

- PostgreSQL running and accessible
- Environment configured so `src.config.Settings` can load (see project root `.env` and `src/config.py`). Alembic imports `settings` in `env.py`, so **all required application env vars** must be set—not only PostgreSQL fields (e.g. app, Redis, and DB settings as defined in config).

## Configuration

- **`alembic.ini`** (project root) — Main Alembic config; the placeholder `sqlalchemy.url` is overridden at runtime in `env.py`.
- **`env.py`** — Sets `sqlalchemy.url` via `get_sync_database_url()`: a synchronous `postgresql+psycopg://` URL built from `settings`, with user and password passed through `urllib.parse.quote_plus` for special characters.
- **`script.py.mako`** — Template for new migration files.
- **`versions/`** — Migration revision scripts.

## Common Commands

Run all commands from the project root:

```bash
# Create a new migration (autogenerate from models)
poetry run alembic revision --autogenerate -m "description of changes"

# Create an empty migration
poetry run alembic revision -m "description of changes"

# Upgrade to latest
poetry run alembic upgrade head

# Upgrade by one revision
poetry run alembic upgrade +1

# Downgrade by one revision
poetry run alembic downgrade -1

# Downgrade all
poetry run alembic downgrade base

# Show current revision
poetry run alembic current

# Show migration history
poetry run alembic history
```

## Workflow

1. **Change models** in `src/models/` (ensure new models are imported so they register on `Base.metadata`).
2. **Generate migration**: `poetry run alembic revision --autogenerate -m "add_xyz_table"`
3. **Review** the generated file in `versions/` and adjust if needed.
4. **Apply locally**: `poetry run alembic upgrade head`

## Deployment

The Docker image runs `alembic upgrade head` in `docker-entrypoint.sh` before starting the API, so a pushed revision is applied when the container starts (same DB credentials as the app).

## Troubleshooting

### "Can't locate revision identified by 'xxx'"

This happens when the database's `alembic_version` table references a revision whose migration file no longer exists (e.g. after deleting or resetting migrations).

**Fix:** Clear the version table so Alembic can start fresh:

```bash
# Example — use your actual DB user, host, and database name:
psql -h "$POSTGRES_HOST" -U "$POSTGRES_USER" -d "$POSTGRES_DB" -c "DELETE FROM alembic_version;"
```

Or using any PostgreSQL client: `DELETE FROM alembic_version;`

Then:

- **Fresh database:** Run `poetry run alembic upgrade head` to apply migrations.
- **Existing schema:** Run `poetry run alembic stamp head` to mark the current migration as applied without running it (only if the live schema already matches that revision).

## Notes

- Always review autogenerated migrations before applying.
- Test downgrades when possible.
- Use descriptive migration messages.
